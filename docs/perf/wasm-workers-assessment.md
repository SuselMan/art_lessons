# Оценка: WebAssembly / Web Workers для движка

## Статус

Исследовательский отчёт (не ADR, ни на что не блокирует). 2026-07-13.

## Вывод коротко

**Смысла сейчас почти нет.** Движок уже не CPU-bound — тяжёлые операции
(смешивание графита, рендер дабов, трансформации слоёв) целиком на GPU через
WebGL1, и это уже сделано эффективно (инстансинг дабов, см. ниже). Всё, что
выполняется на CPU в JS, — это либо копеечная математика на событие указателя,
либо операции, упирающиеся не в скорость вычислений, а в задержку синхронизации
GPU↔CPU (`readPixels`), которую WASM не ускоряет вообще.

Единственный реалистичный кандидат — вынести CPU-скан после `readPixels` в
Web Worker (см. «Что можно сделать» ниже), и то с оговоркой, что сегодня он
вызывается редко и не в горячем пути ввода.

## Как считалось

Прочитан весь `apps/web/src/engine/` (движок — единственное место, где вообще
может быть много вычислений) и `apps/server/src/` (тонкий Socket.io-relay,
сервер ничего не рендерит и не оptre — см. CLAUDE.md). Ниже — разбор по каждому
модулю с реальным объёмом работы.

### DabSystem.ts — сплайн Catmull-Rom → дабы

`_splineDabs()` строит arc-length таблицу на `STEPS=16` шагов и на каждый даб
считает кубический Эрмит + Catmull-Rom интерполяцию скаляров (давление/наклон).
Вызывается **раз на pointermove-событие** (после coalescing — несколько раз на
кадр максимум), а не раз на даб. Даже на быстром росчерке это десятки шагов
чистой арифметики с плавающей точкой — микросекунды. Уже оптимизировано:
переиспользуемые `Float64Array`-скретчи вместо аллокации объектов на каждый
вызов (см. комментарии в файле). WASM здесь не даст выигрыша: стоимость
маршалинга данных на границе JS↔WASM (создание/копирование буферов на каждый
pointermove) превысит экономию на самих вычислениях.

### HapticGrain.ts — хэш-решётка вибрации

Шаг по полуячейкам вдоль сегмента (`Math.ceil(dist / step)` итераций),
`hash2()` — целочисленный хэш. Это уже помечено в коде как «Experimental for
fun prototype», не production-фича. Ничтожный объём работы, не кандидат.

### PointerInput.ts — нормализация событий

Чистая арифметика на событие (`Math.hypot`, аффинное преобразование через
переданный closure). Не измеримо.

### AccumulationBuffer / TiledLayerBuffer / affine.ts — GPU-оркестрация

Это обёртки над WebGL-вызовами (`bindFramebuffer`, `texImage2D`,
`drawArrays`) и композиция 2×3 аффинных матриц (6 умножений). Вся тяжёлая
работа — блендинг графита, бake трансформаций — уже происходит в шейдерах на
GPU (`shaders.ts`, `TRANSFORM_BLIT_FRAG`, `DAB_VERT`/`DAB_FRAG`). CPU здесь
только раздаёт вызовы.

Важно: рендер дабов уже прошёл через оптимизацию инстансинга — комментарий в
`engine/index.ts:2194` явно фиксирует, что раньше было «один `gl.drawArrays` +
~9 `gl.uniform*` вызовов **на каждый даб**», а теперь один
`drawArraysInstancedANGLE` на весь батч дабов инсульта. То есть авторы уже
устраняли самое дорогое место в рендер-пути стандартным GPU-приёмом — а не
JS-вычисления, которые стоило бы выносить в WASM.

### `getContentBounds()` и чекпоинты — единственное реальное CPU-узкое место

Два места делают `gl.readPixels()` (перенос текстуры GPU→CPU) и затем
двойной цикл по пикселям:

- `getContentBounds()` (`engine/index.ts:938`) — скан всех резидентных тайлов
  слоя на непрозрачность, чтобы найти bounding box для гизмо трансформации.
  По коду прямо указано: предполагается вызов «once when tool activates», не
  на кадр.
- `_takeCheckpoint()` (`engine/index.ts:1602`) — снимок пикселей для undo;
  уже вынесен с `requestIdleCallback`, чтобы не блокировать момент отрыва
  пера (там же в комментарии зафиксирован **реальный инцидент**: полноканвасный
  `readPixels` подозревается в провоцировании context loss на мобильных —
  GPU watchdog видит слишком длинный синхронный стол).

Это два разных типа стоимости:

1. Сам `readPixels()` — синхронный GPU→CPU перенос, стоимость которого
   определяется **синхронизацией конвейера GPU**, а не объёмом кода, который
   её вызывает. WASM тут ничего не ускорит — это не JS-вычисление.
2. Двойной цикл после переноса (скан `width×height×4` байт на прозрачность) —
   вот это чистый CPU-loop, и для тайла 1024×1024 это ~1M итераций. Именно
   этот кусок теоретически можно (а) увезти в Web Worker, чтобы не блокировать
   главный поток во время скана, и/или (b) переписать на WASM/SIMD для самого
   скана. Но частота вызова (по коду — не на кадр, не на каждое движение
   пера) означает, что даже нынешний JS-цикл, скорее всего, укладывается в
   бюджет незаметно. Профилировать перед тем, как что-то трогать.

### OperationLog.ts — лог операций

Линейные сканы (`for` по `_entries`) на undo/redo/append — O(n) от размера
истории комнаты, не от размера канваса. При реалистичных объёмах (тысячи
операций за урок) это сотни микросекунд, не узкое место.

### apps/server — тонкий relay

`rooms.ts`/`socketHandlers.ts` (265 + 144 строк) — ретрансляция операций между
сокетами, без рендеринга и без тяжёлых вычислений (см. CLAUDE.md: «server
retransmits operations, never renders»). Нет кандидатов для WASM на сервере.

## Что можно сделать, если/когда это станет проблемой

Если профилирование на реальном устройстве (особенно бюджетный Android-планшет,
о котором уже есть заметки в коде про HapticGrain/#91) покажет реальный jank:

1. **Скан `getContentBounds`/checkpoint-пикселей в Worker.** `readPixels()`
   остаётся на главном потоке (нужен GL-контекст), но получившийся
   `Uint8Array` можно передать в Worker как transferable и сканировать там
   асинхронно. Не устраняет стоимость самого `readPixels`, только убирает
   двойной цикл с главного потока. Оценка выигрыша: небольшая, применимо только
   если профиль покажет, что этот скан реально режет кадр.
2. **Архитектурно иной путь для самой проблемы readPixels/context loss** —
   не WASM/Worker в узком смысле, а вынос рендеринга целиком в Worker через
   `OffscreenCanvas` + `transferControlToOffscreen()`, чтобы GPU-синхронизация
   не блокировала поток, обрабатывающий ввод/UI. Это на порядок больше работы
   (весь `engine/index.ts`, 2919 строк, работает с DOM/`HTMLCanvasElement`
   впрямую) и отдельное архитектурное решение, а не мелкая оптимизация —
   заводить только если context-loss на мобильных станет реально мешать
   пользователям, а не как профилактика.
3. **Будущие фичи — не то, что есть сейчас.** Если в проекте появится что-то
   реально CPU-bound — экспорт в векторный формат, офлайн-рендер PDF,
   сжатие/ресайз изображений на клиенте вручную (а не через
   `createImageBitmap`, который уже отдаёт декодирование браузеру),
   ML-сглаживание штриха — вот это кандидаты на WASM с первого дня, и тогда
   стоит пересматривать вывод этого отчёта.

## Почему не WASM «на всякий случай»

Маршалинг данных через границу JS↔WASM (копирование typed arrays на каждый
вызов) — не бесплатный, и для функций, которые сейчас исполняются за
микросекунды на pointermove (DabSystem, PointerInput), этот оверхед может
оказаться дороже самого вычисления. Оборачивать в WASM «чёрный ящик» имеет
смысл, когда: (а) объём данных на вызов большой (сотни КБ+), (б) вызовов мало
относительно объёма данных (амортизация маршалинга), и (в) вычисление
действительно CPU-bound, а не GPU-sync-bound. Ни один текущий модуль движка не
подходит одновременно под все три условия.
